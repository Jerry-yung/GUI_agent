# SMAN-Bench 运行说明

本文说明官方仓库中 **AppAgent 路线**（`appagent/`）的离线评测数据流，以及接入**自有 Agent** 时需要对齐的接口。Mobile-Agent 路线（`mobileagent/`）思路类似，但 prompt 与对话结构不同。

**推荐自有 Agent 入口**：[`agents/test_vlm.py`](agents/test_vlm.py) — Multi-Path 循环复用官方 `utils.py` 动作映射，VLM 通过 [`llm_set/llm.py`](llm_set/llm.py) 调用，每轮输出 **JSON**（非官方四行文本），由代码解析后写入与 `eval.py` 兼容的结果文件。接口细节见 [`接口.md`](接口.md)。

---

## 1. 目录与数据准备

`--data_dir` 下需同时包含：

| 内容 | 示例 | 作用 |
|------|------|------|
| 任务 JSON | `simple_tasks_sample.json` | 指令 + GT 页面路径 `name` |
| 图语料目录 | `QQmusic_graph_8152/` | 每步截图 / HTML / XML + 4 个全局索引 |

图目录命名须以 App 前缀开头，如 `QQmusic_graph_8152`（代码用 `QQmusic_` 匹配）。

图目录内关键文件：

```
QQmusic_graph_8152/
├── all_action_id.json      # 全局动作字符串 → id（双层 JSON 字符串）
├── all_page_id.json        # 页面名 → id
├── all_page_actions.json   # 每页合法动作 id 列表
├── all_triple.json         # [源页, 动作id, 目标页]
└── QQmusic0_34_347/        # 单页
    ├── QQmusic0_34_347-screen.png
    ├── QQmusic0_34_347-html.txt
    ├── QQmusic0_34_347-xml.txt
    └── QQmusic0_34_347.json
```

任务 JSON 单条示例：

```json
{
  "name": "QQmusic0_34_347",
  "task": "分步描述：\n1. ...\n2. ...\n\n简洁任务：\n帮我整理播放列表..."
}
```

- `name`：GT 页面路径，`QQmusic0` → `QQmusic0_34` → `QQmusic0_34_347`（由下划线分段）。
- `task`：多行文本；最后一行常为「简洁任务」；`^\d+\.\s` 行作为分步 `current_task_desc`。

### 1.1 四个索引文件从哪来？

官方 **SMAN-Bench** 与 **MobileBench-v2** 公开包中**不包含**预生成的 `all_action_id.json` 等 4 个文件；**Mobile3M** 的 `*_graph_*.7z` 解压后只有每页目录与 `{page}.json`（含 `action_space`、`history_actions`、`path`）。

需用本地脚本生成索引（仓库外，一般在 `Mobile3M/` 下）：

| 脚本 | 路径示例 |
|------|----------|
| `convert_mobile3m.py` | `Mobile3M/convert_mobile3m.py` |

对**每个**已解压的图目录执行一次，生成 4 个全局 JSON 后再跑推理 / 评测。

### 1.2 两种 ID 模式（`--mode`）

```bash
cd Mobile3M

# 推荐：探索 ID（与 Mobile3M 论文、任务名后缀、eval GT 一致）
python3 convert_mobile3m.py datasets/QQmusic_graph_8152 --mode explore

# 旧逻辑：按目录名字典序扫描，动作字符串首次出现分配 0,1,2,…
python3 convert_mobile3m.py datasets/QQmusic_graph_8152 --mode legacy
```

| 模式 | 动作 `action_id` 含义 | 适用场景 |
|------|----------------------|----------|
| **explore**（默认） | 与采集时一致：子页面名最后一段数字 = 到达该页的边的探索 ID（见 [Mobile3M 论文](https://arxiv.org/abs/2409.14818)） | 与 `task["name"]` 后缀、`eval.py` GT 对齐 |
| **legacy** | 全图按扫描顺序编号，与页面名后缀**无**对应关系 | 仅兼容旧实验；不推荐新跑分 |

**explore 模式规则简述：**

1. 遍历每个节点：若 `history_actions` 非空，用 `页面名.split('_')[-1]` 作为该 `history_actions[-1]` 的探索 ID。
2. 仅出现在 `action_space`、未作为边出现的动作，从 `max(探索ID)+1` 起递增分配（保证 runner 仍能查表）。
3. 写出 `all_triple.json`：`[path[-2], action_id, 当前页]`，与 `history_actions[-1]` 映射一致。

**切换模式后必须：**

1. 对该 App 的 `*_graph_*` **重新执行** `convert_mobile3m.py`。
2. **重新跑推理**再评测——旧结果 txt 里的 `action_id` 与新模式不一致（例如 legacy 下「歌曲队列」可能是 38，explore 下为 34）。

### 1.3 三套编号（勿混用）

| 名称 | 示例 | 说明 |
|------|------|------|
| 图上 **c1 / s1** | `click(c39)` | `actions_generate` + `draw_bbox_multi` 的**当前页**候选下标，与 `action_id` 无固定相等关系 |
| **`action_id`** | `34` | `all_action_id.json` 中动作字符串的全局 id；explore 模式下与到达子页的探索 ID 一致 |
| **任务名后缀** | `QQmusic0_34_347` 中的 `34`、`347` | 页面路径分段，explore 模式下等于对应步的 GT `action_id` |

推理时：模型输出 `click(cN)`（或 `test_vlm` 的 JSON `action.element`）→ 映射为动作字符串 → 查 `all_action_id` 得到写入 txt 的 `action_id`。

## 2. AppAgent Single-Path 完整数据流（原版）

入口脚本：`appagent/appagent_singlepath.py`  
模型调用：`appagent/model.py`（如 `get_openai_response`）  
工具与标注：`utils.py`  
Prompt 模板：`appagent/prompts.py`（`singlepath_task_template`）

### 2.2 其它任务类型（脚本对照）

| SMAN 类型 | 推理脚本 | 结果格式 |
|-----------|----------|----------|
| Single-path | `appagent/appagent_singlepath.py` | 每行 `action_id: action_info: page_name` |
| Multi-path | `appagent/appagent_multipath.py` | 同上，页面由 Agent 动作动态决定 |
| Multi-path（自有 VLM） | **`agents/test_vlm.py`** | 同上；VLM 输出 JSON，映射逻辑与官方 multipath 一致 |
| Ambiguous | `appagent/appagent_ambiguous.py` | 同上 + 澄清问答 |
| Noisy | `appagent/appagent_noisy.py` | 每行一条动作字符串 `click(...)` |

评测函数见 `eval.py`：`single_path_test` / `multi_path_test` / `noise_data_test`。`test_vlm` 结果可用 **`agents/eval_vlm.py`** 做格式化评测输出（逻辑与 `multi_path_test` 一致）。

## 3. 接入自有 Agent：必须对齐的接口

有两种接入方式：**改官方 runner（推荐）** 或 **完全自写循环但输出格式一致**。

### 3.1 方式 A：只替换模型调用（改动最小）

在 `appagent/appagent_singlepath.py` 的 Round 循环中，把 `get_openai_response(...)` 换成你的函数，或在 `appagent/model.py` 新增：

```python
def get_my_agent_response(prompt, images, **kwargs):
  # 调用你的 Agent
  return True, response_text  # 第一个必须是 bool 表示成功
```

**必须对齐：**

| 接口 | 约定 |
|------|------|
| **输入 - 图像** | 与官方一致：传入 `{page}_labeled.png` 路径列表（已标 c/s 编号） |
| **输入 - 文本** | 使用官方 `singlepath_task_template` 拼好的 `prompt`（含任务与分步描述） |
| **输出 - 文本** | 须能被 `parse_explore_rsp` 解析，包含四段且 `Action:` 行合法 |

**Action 行格式（与 `parse_explore_rsp` 一致）：**

```
Observation: ...
Thought: ...
Action: click(c38)          # 或 scroll(s1, "up") 或 input("文本")
Summary: ...
```

| 动作 | Action 示例 | 说明 |
|------|-------------|------|
| 点击 | `click(c5)` | `c` + 图上数字，对应 `click_actions[编号-1]` |
| 滚动 | `scroll(s2, "up")` | 方向为 `"up"|"down"|"left"|"right"` |
| 输入 | `input("hello")` | 双引号包裹文本 |

后续 **动作映射、写 txt** 仍走官方代码（`all_action_ids` 查表），无需改。

**配置：** 在 `config/config.yaml` 增加模型段，`--model_name` 与 key 一致；OpenAI 兼容 API 用 `--model_type OpenAI`。

---

### 3.2 方式 B：自写 Agent 循环（完全自定义推理）

若不用官方 prompt/标注体系，你仍需保证：

**（1）每步输入（与官方等价）**

- 截图（建议用带 c/s 标注图，否则需自己实现 `actions_generate` + `draw_bbox_multi`）
- 任务全文 / 当前子步骤描述
- 可选：html + xml（若你的 Agent 用结构文本而非只看图）

**（2）每步输出 → 图动作 ID**

最终写入的 `action_info` 必须与 `all_action_id.json` 的 **key 字符串完全一致**，例如：

- `click(<p id="..." ...> 文本 </p>)`
- `scroll(([360, 801], [360, 267]))`

可用官方工具从 html/xml 生成候选，再让你的 Agent 选编号或选字符串。

**（3）结果文件（评测硬约束）**

路径：

```
{save_path}/{task_type}{model_name}/{task["name"]}_{任务下标}.txt
```

Single/Multi/Ambiguous 每行：

```
{action_id}: {action_info}: {page_name}
```

示例（explore 索引下 id 与路径后缀一致）：

```
34: click(<img ... description="歌曲队列" ...>): QQmusic0_34
347: click(<img ... description="删除" ...>): QQmusic0_34_347
```

- `action_id`：来自 `all_action_id.json`；失败常用 `-2`（解析/映射失败）、`-3`（未知动作类型）
- `page_name`：Single-path 下为 GT 页名；**Multi-path 下为 Agent 实际到达页名**

**（4）页面推进**

- **Single-path**：第 `k` 步截图固定为 GT 路径第 `k` 页，**不由 Agent 决定下一页**；Agent 只影响当前步动作是否正确。GT 动作 id 来自各子页 `history_actions[-1]`。
- **Multi-path**（`appagent_multipath.py` / `agents/test_vlm.py`）：下一页由 `action_click` 等经 `all_triple` **动态跳转**；结果 txt 中 `page_name` 为实际轨迹页名。

---

### 3.3 方式 C：替换为 Mobile-Agent 风格

使用 `mobileagent/mobileagent_singlepath.py` + `mobileagent/api.py`，prompt 在 `mobileagent/prompt.py`，输出格式与 AppAgent 相同（结果 txt 一致），但对话为多轮 message 结构。自有 Agent 需对齐 `api.py` 中对应 `get_*_response` 的入参/出参。

---

### 3.4 方式 D：`agents/test_vlm.py`（推荐，已实现）

在 **不修改** `eval.py` / `utils.py` / `appagent_multipath.py` 的前提下，用自有 VLM 跑通 Multi-Path 评测。

| 组件 | 路径 | 作用 |
|------|------|------|
| 主循环 | `agents/test_vlm.py` | multipath 循环、写结果、终端美化输出 |
| VLM 配置 | `llm_set/llm.py` | `vlm = ...` 切换模型；密钥从 `SMAN-Bench/.env` 加载 |
| JSON Prompt | `agents/prompts_vlm.py` | 要求 VLM **只输出 JSON** |
| JSON 解析 | `agents/json_action_parser.py` | JSON → `click(cN)` 等价元组 |
| 终端报告 | `agents/console_report.py` | 推理/评测格式化输出 |
| 评测入口 | `agents/eval_vlm.py` | 格式化版 `multi_path_test` |

**数据流（与官方 multipath 对齐）：**

```
任务 JSON + QQmusic_graph_8152
  → actions_generate + current_actions_generate（过滤合法动作）
  → draw_bbox_multi → {page}_labeled_multi.png（1080×2400）
  → VLM（JSON 输出）→ json_action_parser
  → action_click / action_scroll / action_input / back
  → 动态更新 current_page_name（all_triple 跳转）
  → results/{task_type}{model_name}/{name}_{idx}.txt
  → eval_vlm / multi_path_test
```

**VLM 每轮输出 JSON 协议（不要让模型直接输出官方四行格式）：**

```json
{
  "observation": "屏幕上可见...",
  "thought": "下一步应...",
  "action": {
    "type": "click",
    "element": "c5"
  },
  "summary": "已点击歌曲队列图标"
}
```

| `action.type` | 必填字段 | 代码转换后 |
|---------------|----------|------------|
| `click` | `element`: `"cN"` | `click(cN)` |
| `scroll` | `element`: `"sN"`, `direction` | `scroll(sN, "up")` |
| `input` | `text` | `input("...")` |
| `back` | 无 | `back()` |

解析失败或未知动作写入 `action_id = -3`；映射失败写入 `-2`，与官方一致。

**模型配置**：修改 `llm_set/llm.py` 末尾的 `vlm = ...`（如 `LLM_QWEN3_5_FLASH()`），在 `.env` 配置对应 API Key。**无需** `config/config.yaml`。

**结果目录命名**：`{save_path}/{task_type}{model_name}/`，其中 `model_name` 来自 `get_vlm_model_name()`（如 `qwen3.5-flash`）。示例：`results/multi_testvlmqwen3.5-flash/`。

**Multi-path 结果文件中 `page_name`**：写入 Agent **实际到达**的页面名，不是 GT 路径名（与 `appagent_multipath.py` 相同）。

## 4. 配置与运行示例

### 4.1 依赖（API 模式）

**官方 AppAgent 路线：**

```bash
pip install openai pyyaml opencv-python pyshine anytree colorama tqdm requests pillow numpy
```

**自有 `test_vlm` 额外依赖：**

```bash
pip install langchain-openai langchain-core python-dotenv httpx
```

本地大模型（InternVL 等）另需 `torch`、`transformers`；Mac 上通常无需 `decord`。

### 4.2 配置示例（OpenAI 兼容，如 DashScope）

`config/config.yaml`：

```yaml
qwen3.5-flash:
  OPENAI_API_BASE: "https://dashscope.aliyuncs.com/compatible-mode/v1"
  OPENAI_API_KEY: "sk-xxx"
  OPENAI_API_MODEL: "qwen3.5-flash"
  MAX_TOKENS: 300
  TEMPERATURE: 0.0
  REQUEST_INTERVAL: 10
```

### 4.3 Single-path 与 Multi-path 的区别（截图与指令）

两种模式都会在 prompt 里同时给 **总任务**（`task` 最后一行「简洁任务」）和 **当前步子任务**（`1. …`、`2. …`）。差别在 **下一步看哪张图**：

| 对比项 | Single-path（`appagent_singlepath.py`） | Multi-path（`appagent_multipath.py` / `agents/test_vlm.py`） |
|--------|----------------------------------------|----------------------------------------------------------------|
| 下一步页面 | **固定**：只按 `task["name"]` 的 GT 路径走（如 `QQmusic0` → `QQmusic0_34` → …） | **不固定**：由第一步 action 经 `all_triple` 查到的下一页决定 |
| 第二步截图 | **与第一步 action 无关**，永远是 GT 路径上第 2 页 | **会随第一步 action 变化**；点错可能进错页或卡住 |
| 模型作用 | 只影响「当前页上动作对不对」 | 影响「能否沿图走到目标页 `task["name"]`」 |
| 结束条件 | 走完 `name` 里固定步数 | `current_page_name == task["name"]` 或达到 `--max_rounds` |
| 结果里页面列 | 每步结束后的 **GT 页名** | 每步结束后的 **Agent 实际所在页名** |
| 强依赖索引 | `all_action_id` + `all_page_actions` | 另需 **`all_triple`**（图转移） |

**指令是否按步切换：** 两种模式都会。第 k 轮一般把 `multi_task_desc[k-1]`（即第 k 条 `1./2./…`）填入 `<current_task_desc>`；步数多于分步条数时重复用最后一条。

**分步描述与 Single-path：** 第 1 步给 `1. …`，第 2 步给 `2. …`，但截图仍是 GT 第 2 页，**不是**「按模型上一步点到了哪一页再换图」。

**Multi-path 举例：** 任务终点 `QQmusic0_34_347`。若第 1 步应点「歌曲队列」却点了「播放」，下一步会加载 **播放相关页** 的截图，而不是 GT 的 `QQmusic0_34`。

**标注图差异：** 官方 multipath 与 `test_vlm` 均使用 `{page}_labeled_multi.png`（1080×2400），且 multipath 必须通过 `current_actions_generate` 按 `all_page_actions.json` 过滤当前页合法动作；Single-path 使用 `{page}_labeled.png`。

## 7. SMAN-Bench 测评方式详解 - 跳转逻辑

现在结合代码和论文，我可以给出准确而完整的分析了。

---

### 1. 每个任务是否有多个子指令？

**没有传统意义上的多个子指令。** 每个任务只有一条顶层指令（instruction），比如 "Help me find Beijing's air quality index for tomorrow on Caiyun Weather"。但这条指令内部通过 **slot（槽位）** 蕴含了多个关键信息点。

slot 的作用是充当"子检查点"，比如指令 "I want a 16GB+512GB MacBook Pro M4 in the Midnight version" 对应三个 slot：`16GB+512GB`、`Pro M4`、`Midnight version`，每个 slot 绑定到轨迹中对应的某一个 GUI 页面，在 offline 评测中每经过一个关键页面就触发一次 step reward。

因此可以理解为：**一条指令 + 多个分布在轨迹节点上的 slot 检查点**，而不是多条显式子指令。

---

### 2. 下一个页面是如何确定的？——Single-path vs Multi-path 截然不同

#### Single-path 模式（离线静态）

这是一个完全 **静态、预先固定** 的评测方式。核心逻辑：

- 数据集中预先存储了完整的黄金轨迹（golden trajectory），即 `[截图1 → 动作1 → 截图2 → 动作2 → ...]` 的序列。
- 每个时间步，VLM 拿到的是**预设的、固定的当前截图**（不随 VLM 的动作改变）。
- VLM 输出动作后，eval 脚本将其与 golden action 进行对比：Action ID 是否匹配、动作类型是否正确、坐标是否落在正确区域。
- **不管 VLM 输出什么，下一步永远给它黄金截图（即黄金轨迹的下一帧）**，而非根据 VLM 的动作跳转到实际对应的页面。

从 `eval.py` 的 `single_path_test` 函数可以清晰看到：它读取 golden `real_path`（一组 action ID 序列），再读取 VLM 的预测结果文件，逐步对比 `path == pred_action_num`，完全没有动态页面跳转的逻辑。

```
for i, path in enumerate(real_path):
    pred_action_num = pred_path[i].split(':')[0]
    if path == pred_action_num:
        true_action_sum += 1
```

**结论：Single-path 下，下一页截图是固定的，与 VLM 的输出动作无关。**

---

#### Multi-path 模式（图结构动态跳转）

这是 SMAN-Bench 的核心创新，真正实现了"动作决定下一页"。核心逻辑：

- 底层数据来自 Mobile3M 的有向图结构，节点是 UI 页面，边是 action（带 action_id）。
- `utils.py` 中的关键函数 `action_click` / `action_scroll` / `action_input` 实现了这一机制：

```python
def action_click(click_action, area_idx, actions_list, all_action_ids, current_page_name, all_page_convert):
    action_info = actions_list[click_action[area_idx - 1]]
    click_action = 'click(' + action_info + ')'
    action_id = all_action_ids[click_action]
    new_page_name = all_page_convert[current_page_name + 'act' + str(action_id)]
    return new_page_name, click_action, action_id
```

流程如下：
1. 当前页面 `current_page_name` + VLM 选择的 action（click/scroll/input）→ 查 `all_page_convert` 字典 → 得到 `new_page_name`。
2. 用 `new_page_name` 索引对应的截图和 XML，作为下一步 VLM 的输入。
3. 如果 VLM 选的 action 不在当前页面的合法 action 集合中，则留在当前页面并返回 ERROR（`return current_page_name, "ERROR", -2`）。

**结论：Multi-path 下，下一页截图由 VLM 的动作动态决定**。VLM 点击 c3 就跳到页面 X，点击 c5 就跳到页面 Y，完全真实模拟了在线交互的逻辑，只是所有页面都是离线预存的图结构，而不是真实设备运行的结果。

---

### 两种模式对比总结

| 维度 | Single-path | Multi-path |
|---|---|---|
| 下一页来源 | 固定的黄金截图序列 | VLM 动作动态决定（图跳转） |
| VLM 输入 | 预设截图（与自身输出无关） | 上一步自己选择的动作对应的真实后继页面 |
| 评分逻辑 | 逐步对比 action ID 是否与 golden 一致 | 检查是否经过关键 slot 节点，是否最终到达终止页面 |
| 主要指标 | Step.Acc、Type Acc | SR（成功率）、SE（步骤效率）、Step.Acc（在关键节点处） |
| 优缺点 | 稳定可复现，但不承认等效解法 | 更公平，允许多条等效路径，但需要预构建完整图 |
